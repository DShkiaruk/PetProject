# Kartka — сімейний медичний щоденник

> Project brief та інструкції для роботи над продуктом. Цей файл — джерело
> правди про продукт, його межі й технічні рішення. Тримати в актуальному стані.

## 1. Що це

**Kartka** — приватний сімейний медичний щоденник для українських сімей.
Одне місце для всієї медичної інформації родини. **AI не використовується.**

**Мета:** прибрати хаос із паперами й ризик «загубив виписку / забув ліки /
не пам'ятаю, що призначали», і дати змогу за секунди поділитися повним
анамнезом з лікарем.

**Аудиторія:** дорослі, що ведуть здоров'я родини — часто координують догляд
за дітьми та/або літніми батьками. Головне для них: менше паперового хаосу,
приватність, готовність показати все лікарю за секунди.

**Платформа:** мобільний веб / PWA (mobile-first, встановлюється на домашній
екран без app store), далі адаптив до десктопу.

**Формат проєкту:** особистий портфоліо-проєкт. Фокус на якості UX, UI та чистій
архітектурі. Без публічного запуску, платежів і масштабування зараз — але
архітектуру й приватність робимо по-дорослому.

## 2. Головний продуктовий принцип

Дані вводяться **вручну** — тому введення має бути максимально швидким і
безболісним:

- мінімум полів;
- розумні значення за замовчуванням;
- шаблони;
- повтор останнього запису.

**Якщо введення дратує — продуктом не користуватимуться.** Це критерій №1 при
будь-якому дизайн-рішенні: спершу питати «скільки тапів і полів це коштує
користувачу».

## 3. Ключові фічі (MVP)

1. **Профілі членів сім'ї** + базова картка: алергії, хроніка, група крові.
2. **Документи** — виписки, результати, файли як прикріплені фото/файли з
   кількома ключовими полями (дата, лікар, тип). **Без AI-розбору.**
3. **Трекінг ліків** по кожному члену сім'ї: що приймає, дозування, період,
   історія прийому. Видно, хто що приймає, щоб уникати конфліктів.
4. **Трекінг аналізів** — результати з датою (+ прикріплений файл), історія за
   параметром у часі. Модель — **гібрид**: каталог популярних показників
   (гемоглобін, глюкоза тощо) з одиницями + можливість додати свій параметр.
   Графіки в часі й автодоповнення. **Без референсних норм** (щоб не бути
   медпродуктом).
5. **Нагадування про прийоми до лікарів**, зокрема повторювані (планові чекапи,
   щорічні огляди, стоматолог раз на пів року тощо).
6. **Експорт анамнезу в PDF** — компактний документ для одного члена сім'ї:
   хроніка, алергії, поточні ліки, останні аналізи, історія візитів. Для
   швидкого шерінгу з лікарем.
7. **Екстрений екран** — усе критичне на одному екрані. Тут же — кнопка
   **«Знайти лікарню / травмпункт поряд»** із переходом у зовнішні мапи
   (Google/Apple Maps): зовнішній handoff, **без передачі даних пацієнта** (лише
   загальний геозапит на кшталт «лікарня/травмпункт»), **без тріажу й оцінки**
   «куди краще». Власний каталог закладів не будуємо.

## 4. Межі продукту (тримати з 1-го дня)

- **Це НЕ медичний продукт.** Жодних діагнозів, порад чи висновків — лише запис
  і організація даних. Ніякої інтерпретації показників, ніяких референсних норм.
- **Дані особливої категорії** (чужі, родичів): шифрування ще до запису в базу,
  згода членів сім'ї, відповідність GDPR / закону про захист даних.
- **Жодних зовнішніх AI-сервісів** і відправки медданих третім сторонам.
  Уточнення: зовнішні **переходи-handoff** (напр. «лікарня поряд» → мапи)
  дозволені, **якщо не передають даних пацієнта** — лише загальний геозапит.
  Це не суперечить забороні на відправку медданих третім.

## 5. Дані та приватність

- **Модель зберігання:** хмара з шифруванням at-rest (ключі на сервері).
  Обрано свідомо замість E2EE: для родинного шерінгу й — головне — для надійного
  відновлення доступу. Для медичного архіву «втрата ключа = втрата даних»
  неприйнятна (дані можуть бути критично потрібні саме тоді, коли доступ
  втрачено).
- **Ізоляція доступу:** через row-level security (RLS) на рівні БД, **НЕ** через
  логіку в застосунку. Кожен користувач/родина бачить лише свої дані.
  **Правило: жоден доступ до даних не покладається на `where` у коді — тільки на
  RLS-політики.**
- **Файли** (виписки, результати) зберігаються з тими самими RLS-правилами
  доступу, що й рядки БД.
- **Готовність до E2EE:** шар даних спроектувати так, щоб у майбутньому можна
  було додати E2EE для найчутливіших полів без переписування. Поля-кандидати на
  E2EE позначити, але саме E2EE поки **НЕ** реалізовувати.
- Жодних зовнішніх AI-сервісів і відправки медданих третім сторонам.

## 6. Технічний стек

- **Frontend + серверні хендлери:** Next.js (App Router), деплой на **Vercel**.
- **Backend:** **Supabase** — Postgres (з RLS-ізоляцією по родинах), Auth
  (сімейні акаунти), Storage (файли виписок/аналізів). Усі три з однієї
  інтегрованої точки.
- **Доступ до даних** — через клієнт Supabase (передає контекст користувача в
  RLS-політики). **Prisma поки не додавати**; розглянути пізніше лише за реальної
  потреби в складній типізації (з урахуванням, що Prisma не застосовує RLS
  автоматично).
- **Auth:** email + пароль. Один акаунт на домогосподарство, усередині — кілька
  профілів (діти, батьки). Multi-user/запрошення — поза скоупом MVP, але модель
  даних не має цьому заважати.
- **PWA:** manifest + service worker, встановлення на домашній екран.
- **Нагадування:** web push; врахувати обмеження iOS (push у PWA працює лише
  після встановлення на домашній екран) і закласти запасний варіант (локальні
  нагадування / email).
- **PDF-анамнез:** генерація на сервері (Next.js route handler) зі збережених
  даних, локально, **без AI**.
- **Мови:** українська (основна) + англійська. Закласти i18n-структуру з 1-го
  дня.

## 7. Дизайн-напрям

- **Тон:** теплий, сімейний. Спокійні теплі кольори, м'які форми, людяна мова.
- **Чому:** знижує тривогу навколо медтеми; пасує «сімейному щоденнику», а не
  холодному клінічному інструменту.
- Mobile-first завжди; десктоп — адаптив поверх.

## 8. Робочі правила для розробки

- Кожне дизайн-рішення міряти «вартістю введення» (тапи/поля) — див. розділ 2.
- Доступ до даних — лише через RLS, ніколи через `where` у коді (розділ 5).
- Жодного AI й жодної відправки медданих третім сторонам — без винятків.
- Нічого, що інтерпретує дані як медичну пораду (розділ 4).
- Позначати поля-кандидати на майбутній E2EE, не реалізовуючи його зараз.
